# project-governance Specification

## Purpose

Keep this ML repository governed by OpenSpec so model, calibration, and export
behavior remains explicit and synchronized with the consuming demo app.
## Requirements
### Requirement: OpenSpec project configuration
The repository SHALL contain an `openspec/config.yaml` file that describes the
project purpose, consuming app contract, acceptance targets, and artifact rules.

#### Scenario: OpenSpec validates the project
- **GIVEN** a developer is in the repository root
- **WHEN** they run `openspec validate --all --strict`
- **THEN** OpenSpec validates all specs and active changes without errors

### Requirement: Specs are updated with behavior changes
The project SHALL update the relevant OpenSpec spec or active change for every
non-trivial change to dataset handling, training, calibration validation, or
mobile export before the work is considered complete.

#### Scenario: Training behavior change carries a spec update
- **GIVEN** a developer changes model output decoding or export metadata
- **WHEN** the change is submitted for review
- **THEN** the matching `openspec/specs/mobile-model-export/spec.md` requirement or active change delta is updated

### Requirement: Active changes document implementation intent
Feature work SHALL use an `openspec/changes/<change-name>/` folder containing
`proposal.md`, `design.md`, `tasks.md`, and spec deltas when requirements change.

#### Scenario: New model training phase starts from a change folder
- **GIVEN** the next work is to train a seed-specific segmentation model
- **WHEN** implementation begins
- **THEN** an OpenSpec change exists that describes the training goal, design, tasks, and affected specs

### Requirement: Codex OpenSpec skills are available
The repository SHALL include Codex-facing OpenSpec skills generated by
`openspec init --tools codex` so future coding sessions can use the local
`/opsx:*` workflow.

#### Scenario: Codex skill files exist
- **WHEN** a developer lists `.codex/skills`
- **THEN** OpenSpec skills for proposing, applying, archiving, and exploring changes are present

### Requirement: OpenSpec governs ML repo changes
The project SHALL use OpenSpec as the source of truth for non-trivial changes to
dataset preparation, segmentation training, calibration validation, and mobile
model export behavior.

#### Scenario: New work starts from OpenSpec
- **GIVEN** a developer is about to implement a new training or export behavior
- **WHEN** the behavior changes project capabilities
- **THEN** an active OpenSpec change exists before implementation begins

### Requirement: OpenSpec validation is part of completion
The project SHALL run `openspec validate --all --strict` before considering
OpenSpec-governed work complete.

#### Scenario: Validation catches stale specs
- **GIVEN** a change modifies OpenSpec artifacts
- **WHEN** validation fails
- **THEN** the work is not considered complete until the validation errors are fixed
