# Design — Adopt OpenSpec Governance

## Context

The repo was scaffolded as a Python ML companion to
`advance-seeds-field-inspector-demo`. It already contains helpers for dataset
validation, calibration math, metadata generation, and export-to-app copying,
but no durable spec framework. The user explicitly wants OpenSpec used and
updated throughout the project.

## Decisions

### D1. Canonical specs live under `openspec/specs`

The current scaffold behavior is documented directly as canonical specs because
this is the initial OpenSpec adoption. Future feature work should create active
changes under `openspec/changes/<change-name>/` and update canonical specs when
the change is complete.

### D2. Capability names match ML responsibilities

Specs are organized around durable capabilities:

- `project-governance`
- `dataset-preparation`
- `segmentation-training`
- `calibration-validation`
- `mobile-model-export`

This avoids app-centric names and keeps the repo focused on training/export.

### D3. Config captures app handoff contract

`openspec/config.yaml` includes the consuming app path, stable TFLite filename,
metadata destination, and acceptance targets. That keeps future model work from
drifting away from the app runtime contract.

### D4. Codex skills are generated by the OpenSpec CLI

The setup uses `openspec init --tools codex --profile core .` so future Codex
sessions have local `/opsx:*` skills available. The CLI did not create a config
non-interactively, so project config is added explicitly.

## File Layout

```text
openspec/
  config.yaml
  specs/
    project-governance/spec.md
    dataset-preparation/spec.md
    segmentation-training/spec.md
    calibration-validation/spec.md
    mobile-model-export/spec.md
  changes/
    adopt-openspec-governance/
      .openspec.yaml
      proposal.md
      design.md
      tasks.md
.codex/
  skills/
    openspec-*.md
```

## Validation

The change is valid when:

- `openspec validate --all --strict` passes.
- `python3 -m unittest discover -s tests` still passes.
- README documents the OpenSpec workflow.
